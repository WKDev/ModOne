# OneCanvas Gap Analysis: 전문 EDA 도구 대비 부족한 부분

## Context

ModOne의 OneCanvas를 KiCad, Falstad, Fritzing, EPLAN, OrCAD 등 전문 EDA/회로 설계 도구와 비교하여 부족한 부분을 식별하고, ModOne의 타겟 도메인(PLC/산업 자동화 회로 시뮬레이션)에 맞게 우선순위를 매긴 분석 결과.

---

## ModOne의 고유 강점 (이미 잘하고 있는 것)

| 강점 | 상세 | 경쟁 도구에 없는 것 |
|------|------|---------------------|
| **네이티브 PLC/Modbus 통합** | RTU/TCP 실시간 통신, 캔버스↔PLC 동기화 | KiCad, Falstad, Fritzing 모두 없음 |
| **래더 로직 에디터 통합** | 접점/코일/타이머/카운터 편집 + 회로 캔버스 병행 | 5개 도구 중 없음 |
| **PLC 시뮬레이션 엔진 (Rust)** | 스캔 사이클 시뮬, 브레이크포인트, 스텝 실행 | Falstad는 아날로그만, PLC 전용 없음 |
| **시나리오 기반 테스트** | 시간 기반 이벤트로 자동화 테스트 | 5개 도구 모두 없음 |
| **캔버스 내 오실로스코프** | 4채널 스코프 + Rust 파형 엔진 | Falstad만 부분적 |
| **캔버스↔래더 동기화** | 회로도와 래더가 Modbus 메모리 공유 → 폐루프 시뮬레이션 | 완전히 고유 |
| **VSCode 스타일 IDE 레이아웃** | 탭 에디터, 플로팅 윈도우, 패널, 커맨드 팔레트 | EPLAN만 유사 |

**핵심**: PLC 비주얼 회로 + 래더 로직 + Modbus + 시나리오 테스트의 조합은 어떤 도구도 제공하지 않는 ModOne만의 영역.

---

## Priority 1: Critical (핵심 UX) — 일상 사용에 필수

### 1.1 산업용 컴포넌트 라이브러리 확장
- **현재**: 6개 블록 (PowerSource, LED, Button, PlcOut, PlcIn, Scope)
- **필요**: 릴레이/접촉기(K), 모터 스타터, 퓨즈/차단기, 셀렉터 스위치, 비상정지, 솔레노이드 밸브, 근접/광전 센서, 단자대, 변압기, 파일럿 램프
- **참고 도구**: EPLAN (1000+심볼), KiCad, OrCAD
- **이유**: PLC 회로에는 반드시 접촉기(K1,K2), 과부하 릴레이(F1,F2), 퓨즈가 포함됨. 없으면 실제 PLC 배선도를 그릴 수 없음
- **복잡도**: Medium — `blockDefinitions.ts` 레지스트리 패턴이 확장 가능하게 설계됨
- **핵심 파일**: `types.ts` (BlockType union), `blockDefinitions.ts`, `components/blocks/`

### 1.2 배선 번호/라벨링
- **현재**: Wire에 `id`, `from`, `to`, `color`, `handles`만 있음. 라벨 없음
- **필요**: IEC 60204-1 기준 배선 번호 (L1, N, PE, 101, 102 등)
- **참고 도구**: EPLAN (자동 번호), KiCad (넷 라벨)
- **이유**: 배선 번호 없이는 유지보수 기술자가 사용할 수 있는 도면 생성 불가
- **복잡도**: Small-Medium — Wire 인터페이스에 `label?`, `wireNumber?` 추가 + 렌더러
- **핵심 파일**: `types.ts` (Wire interface), `AnimatedWire.tsx`

### 1.3 크로스 레퍼런스 시스템
- **현재**: 각 블록이 독립적. 코일(K1)과 접점(K1.1, K1.2) 간 연결 개념 없음
- **필요**: 코일 K1의 접점이 어디에 사용되는지 자동 표시
- **참고 도구**: EPLAN (자동 크로스레퍼런스), KiCad (계층 라벨)
- **이유**: 릴레이 로직에서 코일→접점 추적은 기본. 없으면 복잡한 회로 이해 불가
- **복잡도**: Medium-Large — 하나의 디바이스(릴레이)가 여러 블록(코일+접점)을 갖는 모델 필요

### 1.4 캔버스 주석/텍스트 블록
- **현재**: BlockType에 text/annotation 타입 없음. 블록 내부 label만 존재
- **필요**: 자유 텍스트, 타이틀 블록, 섹션 라벨 ("Motor Start Circuit" 등)
- **참고 도구**: 5개 모두 지원
- **이유**: 회로 섹션 구분, 설명 메모, 도면 표제란은 필수
- **복잡도**: Small-Medium — `text` 블록 타입 추가, 시뮬레이션 그래프에서 제외

### 1.5 전기적 규칙 검사 (ERC)
- **현재**: 단락 감지(pathFinder.ts), 개별 연결 검증(connectionValidator.ts) 있으나 종합 검증 없음
- **필요**: 미연결 포트, 플로팅 입력, 전원 충돌, 접지 미연결 등 체계적 검사
- **참고 도구**: KiCad (ERC), OrCAD (DRC), EPLAN
- **복잡도**: Medium — 기존 `circuitGraph.ts`, `pathFinder.ts` 위에 구축
- **핵심 파일**: `circuitGraph.ts`, `pathFinder.ts`, `connectionValidator.ts`

## Priority 2: Important (생산성) — 작업 속도를 크게 향상

### 2.1 PDF/PNG/SVG 내보내기
- **현재**: 내보내기 기능 전무
- **필요**: 인쇄용 PDF (표제란 포함), PNG/SVG 이미지
- **이유**: 산업 프로젝트는 인쇄된 배선도 필수 (승인, 유지보수 매뉴얼, 규정 준수)
- **복잡도**: Medium

### 2.2 넷 라벨 / 버스 연결
- **현재**: 물리적 와이어 연결만 지원. 가상 연결 개념 없음
- **필요**: "+24V" 라벨로 모든 동일 라벨 포인트 전기적 연결
- **이유**: 복잡한 회로에서 와이어 스파게티 방지
- **복잡도**: Medium-Large

### 2.3 정렬/배분 도구
- **현재**: 그리드 스냅(20px)만 있음. 정렬/배분 액션 없음
- **필요**: 선택 블록 좌/우/상/하 정렬, 균등 배분
- **참고 도구**: 5개 모두 + 대부분의 다이어그램 도구
- **복잡도**: Small

### 2.4 찾기/바꾸기 (주소)
- **현재**: SearchPanel.tsx가 하드코딩된 가짜 결과 반환 (mock)
- **필요**: 캔버스+래더 전체에서 디바이스 주소 검색/치환
- **복잡도**: Medium

### 2.5 다중 페이지 회로도
- **현재**: 여러 캔버스 파일 지원하나, 페이지 간 전기적 연결 없음
- **필요**: 페이지 간 크로스레퍼런스, 페이지 번호, 페이지 네비게이터
- **복잡도**: Large

### 2.6 BOM (자재 명세서) 생성
- **현재**: 없음
- **필요**: 컴포넌트 타입, 수량, 주소, 명칭 자동 목록
- **복잡도**: Small — components Map 순회 + CSV/테이블 포맷

### 2.7 컴포넌트/회로 라이브러리
- **현재**: 없음 (클립보드 복사/붙여넣기만)
- **필요**: 재사용 가능한 회로 템플릿 저장/로드 ("모터 기동 회로" 등)
- **복잡도**: Medium

### 2.8 캔버스 미니맵
- **현재**: zoomToFit만 있음
- **필요**: 전체 회로 축소판 + 뷰포트 사각형 + 클릭 네비게이션
- **복잡도**: Medium

## 총평

ModOne은 **PLC/산업 자동화** 영역에서 독보적인 통합 환경을 제공하지만, 전문 도구 대비 **컴포넌트 다양성**(6개 vs 1000+), **문서화 기능**(내보내기/인쇄 없음), **산업 표준 지원**(배선 번호, 크로스레퍼런스 없음)이 가장 큰 갭. Phase 1~2만 구현해도 실무 사용 가능한 수준에 크게 근접할 수 있음.
